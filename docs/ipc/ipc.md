# Nabu IPC Documentation

> **Phase 2.5 — IPC Documentation**
>
> Authoritative reference for the Nabu inter-process communication (IPC) surface.
> This document describes the *finalized* IPC architecture after the Phase 2
> modernization (handler consolidation, preload alignment, error normalization).
>
> It is generated by reading the current implementation and is kept in sync with:
> - `src/shared/channels.ts` — channel enum
> - `src/shared/contracts/index.ts` — request/response/error contracts
> - `src/shared/schemas/index.ts` (re-exporting `src/shared/schemas.ts`) — Zod payload schemas
> - `src/main/ipc/*` — handler modules
> - `src/preload/index.ts` + `src/preload/index.d.ts` — preload bridge
> - `src/shared/ipc/index.ts` — canonical registry

## 1. Overview

### Architecture

Nabu uses Electron's `ipcMain.handle` / `ipcRenderer.invoke` (request–response,
"invoke") and `webContents.send` / `ipcRenderer.on` (push, "send") for all
cross-process communication. The architecture is **layered and contract-driven**:

1. **Shared layer (`@shared`)** — the single source of truth. Defines the
   `IPCChannel` enum, the Zod payload schemas, and the typed `IPCContract`
   objects (request/response/error). No Electron or React imports are allowed
   here.
2. **Main process handlers (`src/main/ipc/*`)** — one module per feature. Each
   module owns exactly its own channels and registers `ipcMain.handle`
   listeners. Handlers validate incoming payloads with the shared schemas and
   normalize all failures through `normalizeError` (see §5).
3. **Preload bridge (`src/preload/index.ts`)** — a thin, type-safe
   `contextBridge` exposing `window.electron`. Every exposed method is a typed
   wrapper around `ipcRenderer.invoke` / `ipcRenderer.on` that derives its
   parameter and return types directly from the shared contracts. It contains
   **no** business logic and **no** duplicated contract definitions.
4. **Renderer (`src/renderer`)** — consumes `window.electron` only; it never
   touches `ipcRenderer` directly.

### Ownership Model

- **Channel ownership** is assigned per *feature module* in the main process.
  There is no overlap: each channel is registered by exactly one
  `register*IPC()` function.
- **Contract ownership** lives in `@shared/contracts` + `@shared/schemas`.
  Handlers and preload both import from there; neither redefines a contract.
- **Service ownership** — handlers delegate to feature services
  (`VaultService`, `SearchService`, `PdfService`, `DictationService`,
  `VectorManager`, `StateManager`, `WidgetManager`, `ClipboardHistory`, etc.)
  constructed once in the bootstrap and threaded through the `IPCContext`.

### Communication Flow

```
Renderer (React component)
   │  window.electron.<feature>.<method>(payload)
   ▼
Preload (contextBridge)
   │  ipcRenderer.invoke(channel, payload)   [or ipcRenderer.on(channel, cb)]
   ▼
IPC Channel (IPCChannel enum value)
   │
   ▼
Handler (src/main/ipc/<feature>.ts → ipcMain.handle)
   │  validates with shared Zod schema
   │  delegates to feature service
   │  normalizes errors via normalizeError()
   ▼
Service (src/main/services/*)
```

For Main→Renderer pushes the flow is reversed: a service or shared helper calls
`sendToRenderer(channel, payload)` (in `src/main/ipc/shared.ts`), which validates
against the outgoing schema map and broadcasts via `webContents.send`. The
renderer subscribes through `window.electron.on.<event>(callback)`.

## 5. Error Contracts

### Canonical normalized error (Phase 2.4)

Every handler routes thrown exceptions through `normalizeError()` in
`src/main/ipc/shared.ts`, producing a `NormalizedError`:

```ts
interface NormalizedError {
  code: string                                  // e.g. "EACCES", "ENOENT", "VALIDATION_ERROR", "HANDLER_ERROR"
  message: string                               // human-readable
  category: 'validation' | 'io' | 'runtime' | 'unknown'
  details?: Record<string, unknown>             // name, truncated stack, context fields
}
```

For channels whose contract `error` field is `z.string()`, the normalized error
is serialized via `errorToString()`:
- `code === 'HANDLER_ERROR'` → raw `message`
- otherwise → `[CODE] message`

This keeps the exact string-typed contract shape the renderer already relies on
while preserving a machine-readable code.

## 7. Registration

### Bootstrap entry — `src/main/ipc/index.ts`

`src/main/ipc/index.ts` is the **single bootstrap entry point** for all IPC
handler registration. It must not contain handler implementations.

Responsibilities:
1. Import the seven feature `register*IPC` functions.
2. Construct the shared `IPCContext` via `createIPCContext(stateManager, vectorManager, watcher)`.
3. Invoke `registerAllIPC(ctx)` which calls each `register*IPC(ctx)` in turn.

```ts
export function createIPCContext(stateManager, vectorManager, watcher): IPCContext {
  return {
    stateManager,
    vectorManager,
    watcher,
    vaultService: new VaultService(stateManager, vectorManager, watcher),
    searchService: new SearchService(stateManager),
    pdfService: new PdfService(),
    dictationService: new DictationService()
  }
}

export function registerAllIPC(ctx: IPCContext): void {
  registerVaultIPC(ctx)
  registerNotesIPC(ctx)
  registerSearchIPC(ctx)
  registerPdfIPC(ctx)
  registerSettingsIPC(ctx)
  registerWidgetsIPC(ctx)
  registerDictationIPC(ctx)
}
```

### `IPCContext` — `src/main/ipc/context.ts`

A plain interface carrying the core managers (`stateManager`, `vectorManager`,
`watcher`) and the instantiated feature services (`vaultService`,
`searchService`, `pdfService`, `dictationService`). It is constructed once in
the bootstrap and threaded to every `register*IPC` function, replacing the
previous monolithic `registerIPCHandlers(stateManager, vectorManager, watcher)`
body.

### Startup registration flow

1. `src/main/index.ts` initializes the core managers (`StateManager`,
   `VectorManager`, `VaultWatcher`).
2. It calls `createIPCContext(...)` to build the shared context.
3. It calls `registerAllIPC(ctx)` once during app startup.
4. Each `register*IPC(ctx)` registers its `ipcMain.handle` listeners (and any
   string-literal widget handlers). Handlers use `ipcMain.removeHandler` where
   needed before `ipcMain.handle` to allow clean re-registration.
5. `setLegacyManagers(...)` and `onWidgetToggle(...)` are re-exported from
   `shared.ts` for backward compatibility during the v1→v2 migration.

### Canonical registry — `src/shared/ipc/index.ts`

`IPC_REGISTRY` (keyed by `IPCChannel` enum) and `IPC_REGISTRY_EXTRA`
(string-literal channels not yet in the enum) form the complete source of truth
for channel → contract mapping. Helpers `getIPCEntry(channel)` and
`isRegisteredChannel(channel)` allow lookup. The registry stores only
contracts and metadata; it does **not** register handlers.

## 9. Maintenance Guidelines

### How to add a channel

1. **Define the contract** in `src/shared/contracts/index.ts` using
   `defineContract({ channel, request, response, error, metadata })`. Reference
   Zod schemas from `src/shared/schemas.ts` (or `src/shared/schemas/index.ts`).
2. **Add the channel constant** to the `IPCChannel` enum in
   `src/shared/channels.ts` (unless it is a temporary string-literal channel,
   which goes in `IPC_REGISTRY_EXTRA` in `src/shared/ipc/index.ts`).
3. **Register the contract** in `IPC_REGISTRY` (or `IPC_REGISTRY_EXTRA`) in
   `src/shared/ipc/index.ts`.
4. **Add the Zod schema(s)** to `src/shared/schemas.ts` if not already present.
5. **Implement the handler** in the appropriate feature module
   (`src/main/ipc/<feature>.ts`) inside its `register*IPC(ctx)` function, using
   `ipcMain.handle(channel, ...)`. Validate with the shared schema and route
   failures through `normalizeError` / `errorToString`.
6. **Expose via preload** in `src/preload/index.ts` (and declare the type in
   `src/preload/index.d.ts`) as a typed wrapper around `ipcRenderer.invoke` /
   `ipcRenderer.on`, deriving types from the contract.

### Where contracts belong

- **Channel identifiers** → `src/shared/channels.ts` (`IPCChannel` enum).
- **Zod payload schemas** → `src/shared/schemas.ts` (re-exported by
  `src/shared/schemas/index.ts`).
- **Typed contracts (request/response/error)** → `src/shared/contracts/index.ts`.
- **Registry entries** → `src/shared/ipc/index.ts` (`IPC_REGISTRY` /
  `IPC_REGISTRY_EXTRA`).
- **No contract logic** may live in handlers or preload; they only import and
  use the shared definitions.

### How preload should be updated

- The preload is a **thin bridge**: never add business logic, validation, or
  duplicated type definitions there.
- Always derive parameter and return types from the shared contracts
  (`Req<T>` / `Res<T>` helpers) so the renderer and main stay in sync.
- For push channels, add a listener under `electron.on.*` that wraps
  `ipcRenderer.on` and returns an unsubscribe closure.

### Handler ownership rules

- Each channel is owned by **exactly one** feature module. Do not register the
  same channel in two modules.
- Handlers must not contain contract definitions; import schemas/contracts from
  `@shared`.
- All exceptions must be normalized via `normalizeError` (Phase 2.4) so the
  channel's declared `error` shape is preserved.
- Use `getSessionForVault(vaultId)` for multi-vault-aware dispatch; fall back to
  the legacy singleton only during migration.
- Use `sendToRenderer(channel, payload)` for Main→Renderer pushes; it validates
  against the outgoing schema map in `shared.ts`.

### Feature module organization

- One file per feature under `src/main/ipc/`: `vault.ts`, `notes.ts`,
  `search.ts`, `pdf.ts`, `settings.ts`, `widgets.ts`, `dictation.ts`.
- Shared helpers live in `src/main/ipc/shared.ts` (never re-implement
  `normalizeError`, `emitActivityLog`, `sendToRenderer`, frontmatter helpers).
- The bootstrap (`src/main/ipc/index.ts`) only constructs `IPCContext` and calls
  `registerAllIPC`; it contains no handler bodies.
- Feature services live under `src/main/services/` and are injected via
  `IPCContext`.

  `widget:show-clipboard` are registered as string-literal handlers but are not
